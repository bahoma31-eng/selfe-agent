# مراجعة كود ملف `agent.py`

## ملخص وظيفة كل دالة

-   **`detect_temperature(message: str) -> float`**:
    تحدد قيمة `temperature` لنموذج الذكاء الاصطناعي بناءً على محتوى رسالة المستخدم. تستخدم كلمات مفتاحية لتصنيف الطلب إلى "كود/تصحيح" (0.2)، "إبداعي" (0.9)، "حقيقي/تحليلي" (0.5)، أو "عام" (0.7).

-   **`load_skills(filepath: str) -> dict[str, str]`**:
    تقرأ ملف `skills.txt` (الذي يحتوي على `skill_name | raw_github_url`) وتُعيد قاموسًا يربط اسم المهارة برابط GitHub الخام الخاص بها. تتجاهل الأسطر الفارغة أو التي تبدأ بعلامة `#`.

-   **`fetch_skill(url: str) -> str`**:
    تجلب محتوى ملف `SKILL.md` من رابط GitHub خام باستخدام مكتبة `urllib.request` المدمجة. تُضيف `User-Agent` مخصصًا وتُطبق مهلة (timeout).

-   **`activate_skill(skill_name: str, skills: dict[str, str], history: list[dict]) -> tuple[str, list[dict]]`**:
    تُفعّل مهارة محددة. تتحقق من وجود المهارة، ثم تجلب محتواها من GitHub عبر `fetch_skill`، وتضبط هذا المحتوى كرسالة نظام (system message) جديدة في سجل المحادثة. تُعيد رسالة تأكيد أو خطأ مع سجل المحادثة المحدّث.

-   **`load_system_prompt(filepath: str) -> str`**:
    تقرأ ملف `system_prompt.txt` وتُعيد محتواه كبرومبت نظام، مع استخدام برومبت افتراضي ("أنت مساعد ذكاء اصطناعي مفيد ودقيق.") إذا كان الملف غير موجود أو فارغ. تتجاهل الأسطر التي تبدأ بعلامة `#`.

-   **`load_keys() -> dict[str, list[str]]`**:
    تحمل مفاتيح API من متغيرات البيئة (مثل `GEMINI_API_KEY_1`، `GROQ_API_KEY`) بناءً على `PROVIDER_CONFIG`. تتحقق من وجود المفاتيح وتُصدر تحذيرًا للمفاتيح المفقودة، وتُنهي البرنامج إذا لم يتم العثور على أي مفتاح لمزود.

-   **`get_key(provider: str, all_keys: dict) -> str`**:
    تُعيد مفتاح API الحالي لمزود معين من القاموس `all_keys`، مع دعم تدوير المفاتيح باستخدام المؤشر `_key_index`.

-   **`rotate_key(provider: str) -> None`**:
    تُحدّث مؤشر المفتاح للمزود المحدد لتدوير المفاتيح، مما يسمح باستخدام مفتاح مختلف في كل طلب (إذا كان هناك عدة مفاتيح متاحة).

-   **`load_models(filepath: str) -> list[dict]`**:
    تقرأ ملف `models.txt` وتُعيد قائمة بالنماذج المتاحة، مع اسم النموذج والمزود الخاص به. تُنهي البرنامج إذا لم يتم العثور على الملف، وتتجاهل الأسطر الفارغة أو التي تبدأ بعلامة `#`، وتتجاهل المزودين غير المعرفين في `PROVIDER_CONFIG`.

-   **`select_model(models: list[dict]) -> dict`**:
    تعرض قائمة بالنماذج المتاحة للمستخدم في واجهة نصية بسيطة وتسمح له باختيار نموذج عن طريق إدخال رقمه. تُكرر الطلب حتى يتم إدخال رقم صحيح.

-   **`chat(model_info: dict, all_keys: dict, user_message: str, history: list[dict]) -> str`**:
    (مقتطعة في الكود المقدم) هذه الدالة هي المسؤولة عن إرسال الرسائل إلى نموذج الذكاء الاصطناعي. من التعليقات، يتضح أنها تتضمن منطقًا لضبط `temperature` ديناميكيًا (باستخدام `detect_temperature`)، وتحديد `max_tokens=4096` لمنع قطع الردود، وآلية إعادة المحاولة (retry) مع `Exponential Backoff` لزيادة المرونة.

## نقاط القوة

1.  **تصميم معياري وواضح**: الكود مقسم إلى دوال صغيرة وواضحة، كل دالة لها مسؤولية محددة، مما يسهل القراءة والصيانة.
2.  **إدارة مفاتيح API مرنة**: دعم لمفاتيح API متعددة لكل مزود (Gemini) وآلية لتدوير المفاتيح (`rotate_key`, `get_key`)، مما يعزز الموثوقية ويقلل من مخاطر تجاوز حدود الاستخدام لمفتاح واحد.
3.  **إدارة إعدادات قابلة للتكوين**: استخدام ملفات نصية (`models.txt`, `system_prompt.txt`, `skills.txt`) لإعداد النماذج، البرومبتات، والمهارات، مما يجعل التكوين سهلًا دون الحاجة لتعديل الكود.
4.  **تحديد `temperature` ديناميكي**: دالة `detect_temperature` هي إضافة ممتازة لتحسين جودة استجابات النموذج بناءً على نية المستخدم، مما يؤدي إلى نتائج أكثر دقة للطلبات التقنية وإبداعية للطلبات الإبداعية.
5.  **آلية جلب المهارات من GitHub**: القدرة على جلب محتوى المهارات من روابط GitHub الخام مباشرةً تزيد من مرونة وقابلية توسيع الوكيل، وتسمح بتحديث المهارات بسهولة.
6.  **معالجة الأخطاء الأساسية**: تتضمن الدوال معالجة لعدم وجود ملفات (مثل `models.txt`, `system_prompt.txt`)، ومفاتيح API مفقودة، وأخطاء جلب المهارات، مما يجعل الوكيل أكثر قوة.
7.  **استخدام مكتبات مدمجة**: الاعتماد على `urllib.request` لجلب المهارات يقلل من التبعيات الخارجية.
8.  **واجهة مستخدم بسيطة لاختيار النموذج**: دالة `select_model` توفر طريقة تفاعلية وواضحة للمستخدم لاختيار النموذج المطلوب.
9.  **تحسينات الأداء والموثوقية (مذكورة في التعليقات)**: `max_tokens=4096` لمنع القطع، وآلية `retry` مع `Exponential Backoff` (في دالة `chat`) تُظهر اهتمامًا بتحسين تجربة المستخدم وموثوقية الاتصال بالنماذج.

## نقاط التحسين المقترحة

1.  **توحيد معالجة الأخطاء**: حاليًا، بعض الدوال تُنهي البرنامج مباشرةً (`sys.exit(1)`) عند الأخطاء الحرجة (مثل عدم وجود `models.txt` أو مفاتيح API)، بينما تُعيد دوال أخرى سلاسل نصية للأخطاء (`activate_skill`, `fetch_skill`). قد يكون من الأفضل استخدام نظام استثناءات موحد (custom exceptions) لجميع الأخطاء الحرجة وغير الحرجة، مما يتيح معالجة أكثر مرونة في الدوال العليا.
2.  **تحسين دالة `chat`**: نظرًا لأهمية دالة `chat`، يجب توفير الكود الكامل لها لتحليلها بشكل أعمق. من التعليقات، يبدو أنها تحتوي على منطق مهم (retry, backoff)، وتفاصيل التنفيذ ستكون حاسمة لتقييمها بشكل كامل.
3.  **التحقق من صحة الروابط في `load_skills`**: حاليًا، `load_skills` تقرأ الروابط كما هي. يمكن إضافة تحقق بسيط (مثل regex) للتأكد من أن الروابط تتبع صيغة URL صحيحة، أو على الأقل صيغة GitHub raw URL المتوقعة.
4.  **تحسين رسائل الخطأ للمستخدم**: بعض رسائل الخطأ (مثل `[ERROR] لا يوجد مفتاح صالح للمزود '{provider}'.`) موجهة للمطور. يمكن تحسينها لتكون أكثر وضوحًا وإرشادًا للمستخدم النهائي، أو فصل رسائل الخطأ الداخلية عن تلك التي تُعرض للمستخدم.
5.  **إدارة حالة `_key_index`**: المتغير `_key_index` هو متغير عام. في تطبيقات أكبر أو متعددة الخيوط، قد يؤدي هذا إلى مشاكل. يمكن تضمينه في كلاس أو تمريره كجزء من كائن حالة إذا كان الوكيل سيتطور ليصبح أكثر تعقيدًا.
6.  **التعامل مع `User-Agent` في `fetch_skill`**: استخدام `User-Agent` مخصص جيد، ولكن يمكن جعله قابلاً للتكوين (مثل قراءته من متغير بيئة أو ملف إعدادات) بدلاً من أن يكون ثابتًا في الكود.
7.  **توسيع `detect_temperature`**: يمكن تحسين دقة `detect_temperature` باستخدام نماذج لغوية صغيرة لتصنيف نية المستخدم بشكل أكثر تعقيدًا بدلاً من مجرد الكلمات المفتاحية، أو إضافة خيار للمستخدم لتجاوز التحديد التلقائي.
8.  **التوثيق الإضافي**: على الرغم من وجود تعليقات جيدة، يمكن إضافة docstrings لجميع الدوال تشرح الغرض، المعلمات، والقيم المُعادة بشكل أكثر تفصيلاً، خاصة للدوال المعقدة مثل `chat` (عند اكتمالها).
9.  **اختبارات الوحدة (Unit Tests)**: لا يوجد ذكر لاختبارات الوحدة. كتابة اختبارات لكل دالة (خاصة `detect_temperature`, `load_skills`, `load_keys`, `activate_skill`) ستضمن استقرار الكود وتسهل عمليات التطوير المستقبلية.
